# 🚀 출시 직전 기준 문서 (운영 규칙)

**목적**: 출시 직전 상태 UI, 타이밍, Debug 분리 규칙 정리  
**대상**: 개발자 / QA / 운영팀  
**상태**: 출시 준비 완료 기준

---

# ✅ 1️⃣ 상태 UI 단일화 규칙

## 🎯 핵심 원칙

> **한 순간에 사용자에게 보이는 "상태 메시지는 단 하나"**

---

## 상태 UI 종류 정의

### A. 에러 상태 (최우선)

**특징**: 치명적 문제, 사용자 액션 필요

**예시**:
- 네트워크 실패
- 외부 지도 열기 실패
- 검색 실패

**UI 패턴**:
```
📡 지금은 연결이 불안정해요
[다시 시도] [나중에]
```

**규칙**: 다른 모든 상태 UI를 덮어씀

---

### B. 로딩 / 처리 중 상태

**특징**: 진행 중, 완료 대기

**예시**:
- 검색 중
- AI 추론 중
- 위치 계산 중

**UI 패턴**:
```
🔍 찾는 중…
```

**규칙**: 화면에 단 하나만 표시

---

### C. 컨텍스트 설명 상태

**특징**: 추가 정보, 선택적

**예시**:
- "○○ 근처에서 찾는 중"
- "현재 위치 기준"

**UI 패턴**:
```
"축구장" 근처에서 찾는 중…
```

**규칙**: 로딩이 끝난 뒤에만 노출

---

### D. Idle 상태 (기본)

**특징**: 아무것도 없음, 대기 중

**UI 패턴**: 없음 (기본 화면)

---

## ❌ 절대 금지 규칙

| 금지 사항 | 이유 |
|----------|------|
| 로딩 + 컨텍스트 동시 표시 | 사용자 혼란 |
| 에러 + 로딩 동시 표시 | 상태 충돌 |
| Debug UI + 운영 UI 동시 표시 | 전문성 저하 |

---

## 상태 우선순위 (엄격히 지킴)

```
1. Error (에러)
2. Loading (로딩)
3. Context (컨텍스트)
4. Idle (기본)
```

**규칙**: 위 순서대로 우선순위 적용, 상위 상태가 하위 상태를 덮어씀

---

## 코드 규칙 예시

```tsx
// ✅ 정답 구조
if (error) return <ErrorBanner />;
if (isLoading) return <LoadingPill />;
if (context) return <ContextHint />;
return null; // Idle
```

**핵심**: 이 구조를 **모든 상태 UI에 공통 적용**

---

## 실전 체크리스트

- [ ] 에러 상태가 다른 상태를 덮어씀
- [ ] 로딩 상태는 화면에 단 하나만
- [ ] 컨텍스트는 로딩 완료 후에만
- [ ] 상태 UI가 동시에 표시되지 않음

# ✅ 2️⃣ 로딩 / 검색 / 완료 UX 타이밍 설계

## 🎯 핵심 원칙

> **"지금 뭐 하는 중인지"를 사용자가 추측하지 않게 만든다**

---

## 단계별 타이밍 정의

### ① 사용자 액션 발생

**트리거**:
- 마커 클릭
- 검색 버튼 클릭
- 음성 명령

**즉시 표시**:
```
🔍 찾는 중…
```

**지연 시간**: **0ms** (즉시)

---

### ② 처리 중 (AI / 검색 / 계산)

**상태 유지**:
- 로딩 UI 계속 표시
- 애니메이션 단순하게 (깜빡임 ❌)

**타임아웃**: 최대 10초 (그 이후 에러 상태)

---

### ③ 처리 완료 (성공)

**즉시 전환**:
1. 로딩 UI 제거 (0ms)
2. 결과 UI 표시
3. 필요 시 TTS 1회

**예시**:
```
여기로 안내할게요
```

**규칙**: 로딩 → 결과 사이 공백 시간 ❌

---

### ④ 처리 완료 (실패)

**즉시 전환**:
1. 로딩 UI 제거 (0ms)
2. 에러 상태 진입

**예시**:
```
지금은 찾을 수 없어요
[다시 시도]
```

---

## UX 타이밍 체크리스트

- [ ] 로딩은 **항상 즉시** (0ms 지연)
- [ ] 로딩은 **영원히 남아 있지 않음** (타임아웃 10초)
- [ ] 성공/실패 후 **로딩 UI 즉시 제거** (0ms)
- [ ] 음성은 **완료 시점에만** (1회)
- [ ] 로딩 → 결과 사이 **공백 시간 없음**

---

## 타이밍 다이어그램

```
사용자 액션 (0ms)
  ↓
로딩 UI 표시 (0ms)
  ↓
처리 중... (최대 10초)
  ↓
성공 → 결과 UI + TTS (0ms)
실패 → 에러 UI (0ms)
```

# ✅ 3️⃣ Debug → 운영 환경 완전 분리

## 🎯 핵심 원칙

> **사용자는 "Debug"라는 단어를 절대 보면 안 된다**

---

## Debug UI 정의

다음 중 하나라도 해당하면 **Debug**다:

- "Debug" 라벨
- 내부 상태 값 표시 (예: `{phase: 33}`)
- 테스트 버튼
- 개발자만 이해하는 문구
- 콘솔 로그 과다 출력

---

## 필수 분리 규칙

### 1️⃣ 환경 분기 렌더링

```tsx
// ✅ 정답
{process.env.NODE_ENV === 'development' && <DebugUI />}
```

**규칙**: production 빌드에는 **존재 자체가 없음**

---

### 2️⃣ 스타일 분리

**Debug UI**:
- `z-index` 낮게 (운영 UI 아래)
- 운영 UI와 레이어 공유 ❌
- 명확한 시각적 구분

**운영 UI**:
- `z-index` 정상 (사용자에게 보임)
- Debug UI와 격리

---

### 3️⃣ 콘솔 로그 최소화

**개발 환경**:
```tsx
if (process.env.NODE_ENV === 'development') {
  console.log('[Debug]', data);
}
```

**운영 환경**:
- `console.log` 최소화
- `console.error`만 유지 (필수)
- UX 이벤트 로그는 별도 시스템

---

## 배포 전 체크리스트 (필수)

- [ ] 화면에 "Debug" 글자 없음
- [ ] 테스트 버튼 없음
- [ ] 내부 상태 값 표시 없음
- [ ] 콘솔 로그 최소화
- [ ] QA/심사 스크린샷에 Debug UI 없음
- [ ] production 빌드에서 Debug 코드 제거 확인

---

## 실전 팁 (강력)

> **Debug UI는 "보이게 숨기지 말고, 아예 빌드에서 제거"**

이게 제일 안전하다.

**방법**:
1. 환경 변수로 조건부 렌더링
2. 빌드 시 Dead Code Elimination 활용
3. 배포 전 production 빌드 확인
