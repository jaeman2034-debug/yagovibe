# 🔒 상태 기반 UI 표시 규칙 (Guest vs Member) - STEP 5 LOCK

## 목표

**게스트에게는 '가능성'을 보여주고, 회원에게는 '즉시 실행'을 준다.**
UI는 유혹해야지 차단하면 안 된다.

---

## 1️⃣ UI 설계의 절대 원칙 (천재 기준)

- 버튼은 숨기지 않는다
- 비활성(disabled)은 최소화
- 클릭 → 그 순간에만 막고 연결한다
- "못 합니다" ❌ → "계속하려면 가입" ⭕

---

## 2️⃣ 상태 판별 함수 (UI 단일 진실)

```typescript
const isGuest = getUserState() === "GUEST";
```

👉 **모든 UI 분기는 이 한 줄만 사용**

---

## 3️⃣ 기능별 UI 표시 규칙 (핵심 테이블)

### 🟢 항상 보여준다 (Guest / Member 동일)

| UI 요소 | 이유 |
|---------|------|
| 상품 카드 | 탐색 유도 |
| 상품 상세 버튼 | 가치 경험 |
| 가격/사진 | 신뢰 |
| 채팅 목록(읽기) | 거래 맥락 |
| 판매자 정보 | 안심 |

### 🟡 클릭은 가능, 행동 시 가입 유도

| UI 요소 | 게스트 행동 |
|---------|------------|
| "채팅 보내기" 버튼 | 클릭 → Lazy Signup |
| "판매하기" 버튼 | 클릭 → Lazy Signup |
| "찜" 버튼 | 클릭 → Lazy Signup |
| "거래 요청" | 클릭 → Lazy Signup |

👉 **버튼을 숨기지 않는다가 핵심**

### 🔴 회원 전용 (게스트에겐 미노출)

| UI 요소 | 이유 |
|---------|------|
| 결제 진행 버튼 | 오해 방지 |
| 배송지 입력 | 가입 후 의미 있음 |
| 계정 설정 | 맥락 불필요 |

---

## 4️⃣ 버튼 상태 UX 규칙 (중요)

### ❌ 하지 말 것

- disabled 버튼 + "로그인 필요"
- 회색 처리 + 클릭 불가

### ⭕ 정답 패턴

- 활성 버튼
- 클릭 시 Lazy Signup 모달

```tsx
<button onClick={() => requireAuth(action)}>
  채팅 보내기
</button>
```

---

## 5️⃣ CTA 문구 규칙 (Guest vs Member)

### 같은 버튼, 다른 행동

| 상태 | 버튼 문구 |
|------|-----------|
| Guest | 채팅 보내기 |
| Member | 채팅 보내기 |

👉 **문구를 바꾸지 않는다 → 클릭 결과만 다름**

---

## 6️⃣ 상단/하단 고정 CTA 설계 (모바일 기준)

### 예: 상품 상세 하단

**Guest:**
```
[ 채팅 보내기 ]
```
→ 클릭 시 Lazy Signup

**Member:**
```
[ 채팅 보내기 ]
```
→ 즉시 입력창 포커스

👉 **UI는 동일, 행동만 다름**

---

## 7️⃣ 상태 기반 배너/안내 (최소 허용)

### 허용되는 단 하나의 배너 (선택)

```
지금 가입하면
바로 거래를 시작할 수 있어요
```

❌ 반복 노출
❌ 강제 노출

---

## 8️⃣ 이 설계가 강력한 이유

- 게스트는 하고 싶은 걸 다 본다
- "막힘"이 행동 순간 1회만 발생
- 가입은 "차단 해제"처럼 느껴짐
- UI 분기가 적어 코드/디자인 유지보수 쉬움

---

   → ✅ Guest/Member 동일한 UI, 행동만 다름
