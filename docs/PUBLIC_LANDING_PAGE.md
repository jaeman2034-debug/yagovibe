# 🏛️ 공공용 랜딩 페이지 설계 문서

## 목적

서비스를 설명하지 않고, 기록의 존재만 보여줌

- ❌ 가입 유도
- ❌ 기능 소개
- ❌ 문의 버튼
- ⭕ 기록만 존재

---

## 1️⃣ URL & 접근

```
/public
/public/region/{region}
/public/sport/{sport}
```

**접근 제어**:
- 로그인 ❌
- 추적 스크립트 최소
- 쿠키 배너 ❌ (가능하면)

---

## 2️⃣ 상단 헤더 (톤 고정)

**제목**:
```
지역 팀 활동 기록
또는
{region} 지역 팀 활동 기록
또는
{sport} 팀 활동 기록
```

**보조 문장**:
```
일정 기간 동안 이어진 팀 활동을 기록으로 모았습니다.
```

**제외 사항**:
- ❌ "플랫폼"
- ❌ "서비스"
- ❌ "활용 방법"

---

## 3️⃣ 핵심 콘텐츠 (위→아래)

### ① 최근 기록 스트림 (시간순)

**카드 구성**:
```
[팀명]
소흘 60대 FC

[기록 제목]
7월 셋째 주 정기 모임 기록

[메타]
2025년 7월 · 축구 · 포천
```

**규칙**:
- 최신순
- 클릭 시 개별 포스트 페이지
- CTA 없음

### ② 지역/종목 요약 (아주 작게)

**페이지 하단**:
```
지난 30일간
- 기록 18건
- 팀 7곳
```

**제외 사항**:
- ❌ 그래프
- ❌ 비교
- ❌ 순위

---

## 4️⃣ 빈 상태 처리

**기록 0개**:
```
아직 이 범위에 해당하는 기록이 없습니다.
```

**제외 사항**:
- ❌ 모집 문구
- ❌ "첫 기록"

- ❌ 댓글/반응

**공공 열람 기준 유지**

## 7️⃣ 왜 이 페이지가 필요한가

- **외부 링크 시 오해 없음**
- **공공 협업 시 설명 불필요**
- **서비스 성격이 인프라로 고정**

**핵심 인식**: "이건 홍보가 아니라 기록이다"

---

## 8️⃣ 성공 신호 (정성)

- "보고만 해도 이해되네요"
- "공공 데이터 같아요"
- "설명 필요 없네요"

**내부 기준 문장**: 설명 페이지가 필요 없어질수록, 구조는 완성에 가깝다.

## 🔟 차이점: 아카이브 vs 공공

| 항목 | 아카이브 | 공공 |
|------|---------|------|
| URL | `/archive/...` | `/public/...` |
| 네비게이션 | 있음 (홈으로) | 없음 |
| 목적 | 내부 아카이브 | 외부 공개 |
| 톤 | 기록 집계 | 기록 존재만 |
