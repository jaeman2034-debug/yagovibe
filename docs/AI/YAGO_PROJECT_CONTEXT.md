# YAGO 프로젝트 컨텍스트

**생성일**: 2025-01-27  
**프로젝트 상태**: 프로덕션 운영 중, 리팩토링 단계

---

## 프로젝트 요약

YAGO VIBE SPT는 **AI 기반 통합 스포츠 플랫폼**으로, 스포츠 팀 관리, 중고거래 마켓, 실시간 채팅, 지도 기반 네비게이션, 활동 피드 등의 기능을 제공하는 종합 플랫폼입니다.

### 핵심 가치
- **스포츠 커뮤니티 중심**: 팀 운영, 경기 매칭, 용병 모집
- **실시간 커뮤니케이션**: Firestore 기반 실시간 채팅
- **지리 기반 서비스**: Google Maps 통합, 위치 기반 검색
- **AI 기반 자동화**: 리포트 생성, 추천 시스템, 이상 탐지

---

## 기술 스택

### Frontend
- **프레임워크**: React 19.0.0
- **빌드 도구**: Vite 7.3.1
- **라우팅**: React Router v7
- **언어**: TypeScript 5.9.3
- **스타일링**: 인라인 스타일 + CSS 모듈

### Backend
- **인프라**: Firebase (Hosting, Functions, Firestore, Storage, Auth)
- **Functions**: Node.js (TypeScript)
- **리전**: asia-northeast3 (Seoul)

### 외부 서비스
- **지도**: Google Maps JavaScript API
- **AI/ML**: OpenAI (GPT-4o-mini, Vision API)
- **알림**: Firebase Cloud Messaging (FCM)
- **분석**: BigQuery (리포트 생성)

### 개발 도구
- **패키지 관리**: npm
- **버전 관리**: Git
- **배포**: Firebase Hosting + GitHub Actions

## 핵심 기능

### 1. Market (중고거래 마켓)
- **복잡도**: ⭐⭐⭐⭐⭐ (155개 파일, 60+ Cloud Functions)
- **기능**: 상품 등록, 검색, 거래, 채팅, 검증, 광고, 악용 방지
- **특징**: 
  - AI 기반 이미지 분석
  - 가격 이상 탐지
  - 에스크로 시스템
  - 실시간 피드 엔진

### 2. Chat (실시간 채팅)
- **복잡도**: ⭐⭐⭐⭐ (46개 파일)
- **기능**: 1:1 채팅, 그룹 채팅, 팀 채팅, STT/TTS, 미디어 공유
- **현재 문제**: ChatPage.tsx가 2,520줄로 과도하게 크고 복잡함
- **리팩토링 대상**: 최우선

### 3. Map (지도 및 네비게이션)
- **복잡도**: ⭐⭐⭐ (62개 파일)
- **기능**: Google Maps 통합, 마커 클러스터링, 네비게이션, 음성 인식
- **최근 수정**: 모바일 레이아웃 안정화 완료

### 4. Activity (활동 피드)
- **복잡도**: ⭐⭐ (13개 파일)
- **기능**: 활동 피드, 이벤트, 일정 관리
- **상태**: 상대적으로 단순, 안정적

### 5. Team (팀 관리)
- **기능**: 팀 생성, 멤버 관리, 권한 시스템, 이벤트 관리
- **권한 시스템**: owner, admin, member (팀 레벨) + ADMIN, USER (플랫폼 레벨)

## 현재 위험 요소

### 🔴 Critical
1. **ChatPage.tsx 과도한 복잡도**
   - 2,520줄 단일 파일
   - UI, 비즈니스 로직, Firebase 호출이 모두 혼재
   - 테스트 불가능, 유지보수 어려움
   - 리팩토링 시도 시 버그 위험 높음

2. **Market 도메인 과도한 복잡도**
   - 155개 파일, 60+ Cloud Functions
   - 비즈니스 로직이 여러 레이어에 분산
   - 의존성 추적 어려움

### 🟡 Medium
3. **서비스 레이어 부재**
   - Firebase 호출이 컴포넌트에 직접 분산
   - 재사용성 낮음, 테스트 어려움

4. **타입 정의 불일치**
   - Firestore 스키마와 TypeScript 타입 불일치 가능성
   - 런타임 오류 위험

### 🟢 Low
5. **폴더 구조 비일관성**
   - `src/pages`, `src/components`, `src/features` 혼재
   - 명확한 아키텍처 원칙 부재

## 리팩토링 원칙

### 1. UI/UX 변경 금지
- **절대 규칙**: 리팩토링 중 UI/UX는 변경하지 않음
- 사용자 경험에 영향 없이 내부 구조만 개선
- 시각적 변경은 리팩토링 완료 후 별도 작업

### 2. 구조 우선, 기능 나중
- 먼저 코드 구조를 개선
- 기능 추가/변경은 구조 안정화 후 진행
- 점진적 리팩토링 (Big Bang 방식 금지)

### 3. 안전한 추출 우선
- UI 컴포넌트부터 추출
- 비즈니스 로직은 나중에
- Firebase 호출은 서비스 레이어로 이동

### 4. 테스트 가능성 확보
- 컴포넌트와 로직 분리
- 의존성 주입 가능한 구조
- 단위 테스트 작성 가능하도록

### 5. 점진적 마이그레이션
- 기존 코드와 새 코드 공존 허용
- 한 번에 하나씩 마이그레이션
- 각 단계마다 빌드/동작 확인

---

## 절대 규칙 (Non-Negotiable)

### ❌ 하지 말 것
1. **UI/UX 변경**: 리팩토링 중 UI 변경 금지
2. **Firebase 스키마 변경**: Firestore 구조 변경 금지
3. **한 번에 전체 재작성**: Big Bang 리팩토링 금지
4. **기능 추가**: 리팩토링 중 새 기능 추가 금지
5. **의존성 업그레이드**: 패키지 버전 업그레이드 금지 (리팩토링 중)

### ✅ 반드시 할 것
1. **단계별 검증**: 각 단계마다 빌드/동작 확인
2. **기존 동작 보존**: 현재 동작 100% 유지
3. **점진적 추출**: 작은 단위로 나눠서 추출
4. **문서화**: 변경 사항 문서화
5. **안전한 롤백**: 문제 발생 시 즉시 롤백 가능하도록

## 현재 리팩토링 대상

### Phase 1: ChatPage.tsx 분리 (진행 중)
- **목표**: 2,520줄 → 200줄 이하
- **전략**: UI 컴포넌트 → 훅 → 서비스 레이어 순서로 추출
- **예상 기간**: 2-3주

### Phase 2: 서비스 레이어 구축
- **목표**: Firebase 호출을 서비스 레이어로 이동
- **범위**: Chat, Market, Activity, Team
- **예상 기간**: 1-2주

### Phase 3: Market 도메인 정리
- **목표**: 155개 파일 구조 정리
- **전략**: 기능별 그룹화, 중복 제거
- **예상 기간**: 3-4주
