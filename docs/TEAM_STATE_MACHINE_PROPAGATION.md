# 🚀 상태 머신 구조를 팀에 전파하는 방법

> 이걸 알면:
> - 네가 설계 주도권 가진다
> - 리뷰에서 안 밀린다
> - "왜 이렇게 했냐" 질문이 사라진다

---

## 1️⃣ 문서화 전략 (먼저 해야 할 것)

### 필수 문서

1. **상태 머신 설계 원칙** (1페이지)
2. **구현 가이드** (코드 예시 포함)
3. **QA 체크리스트** (검증 기준)

### 문서 구조

```
docs/
├── STATE_MACHINE_PRINCIPLES.md      # 설계 원칙
├── STATE_MACHINE_IMPLEMENTATION.md  # 구현 가이드
├── STATE_MACHINE_QA_CHECKLIST.md    # QA 기준
└── examples/
    ├── navigation-state-machine.tsx  # 지도 페이지 예시
    ├── search-state-machine.tsx     # 검색 페이지 예시
    └── list-state-machine.tsx        # 리스트 페이지 예시
```

**핵심**: 문서는 **1페이지씩**. 길면 안 읽음.

---

## 2️⃣ 설득 전략 (왜 이렇게 해야 하는가)

### ❌ 잘못된 설득

```
"이게 더 좋아요"
"이렇게 하면 깔끔해요"
```

### ✅ 정답 설득

```
"QA가 이 구조 좋아해요"
"기능 늦어도 UX는 완성돼요"
"팀원 바뀌어도 설명 쉬워요"
```

### 구체적 예시

#### 기획자에게

```
"이 구조면 기능 스펙이 늦어도
UI/UX는 먼저 완성할 수 있어요.
QA도 기능 없이도 테스트 가능해요."
```

#### 개발자에게

```
"이 구조면 API 교체해도 UI 안 망가져요.
상태 머신만 지키면 기능 추가가 쉬워요.
디버깅도 상태만 보면 돼요."
```

#### 디자이너에게

```
"이 구조면 디자인 바뀌어도
코드 구조는 그대로예요.
상태별 UI만 바꾸면 돼요."
```

## 3️⃣ 코드 리뷰 전략 (리뷰에서 안 밀리는 법)

### 리뷰 체크리스트

리뷰어에게 제공:

```markdown
## 상태 머신 리뷰 체크리스트

- [ ] UI 상태와 기능 상태가 분리되어 있는가?
- [ ] 상태별 UI가 명확히 정의되어 있는가?
- [ ] 상태 전환이 사용자 액션으로만 발생하는가?
- [ ] 디버그 배지가 있는가?
- [ ] 지도 페이지와 같은 사고방식인가?
```

### 리뷰 코멘트 템플릿

#### ❌ 잘못된 코드 발견 시

```
이 부분은 상태 머신 원칙에 맞지 않아요.

현재: API 응답이 UI 상태를 직접 변경
권장: UI 상태 머신이 API를 제어

참고: docs/STATE_MACHINE_IMPLEMENTATION.md
```

#### ✅ 좋은 코드 발견 시

```
상태 머신 원칙을 잘 지키셨네요!

UI 상태와 기능 상태가 분리되어 있고,
상태 전환이 명확한 트리거로만 발생합니다.
```

## 4️⃣ 온보딩 전략 (새 팀원 교육)

### 1단계: 지도 페이지 보여주기

```
"이 지도 페이지가 우리 앱의 설계 철학이에요.
다른 페이지도 이렇게 만들 거예요."
```

### 2단계: 상태 머신 설명

```
"모든 페이지는 상태 머신이에요.
UI 상태와 기능 상태를 분리해요."
```

### 3단계: 예시 코드 보여주기

```
"이 코드가 템플릿이에요.
새 페이지 만들 때 이 구조를 따라주세요."
```

### 4단계: 실전 적용

```
"이제 검색 페이지를 이 구조로 만들어볼게요.
함께 해볼까요?"
```

---

## 5️⃣ 점진적 전파 전략 (한 번에 다 바꾸지 말 것)

### Phase 1: 지도 페이지 완성 (이미 완료)

- ✅ 상태 머신 구조 증명
- ✅ QA 통과
- ✅ 문서화

### Phase 2: 검색 페이지 적용

- 새 검색 페이지를 상태 머신으로 설계
- 지도 페이지와 같은 구조
- 팀 리뷰 및 피드백

### Phase 3: 리스트 페이지 적용

- 기존 리스트 페이지 리팩토링
- 상태 머신 구조로 전환
- 점진적 적용

### Phase 4: 앱 전체 확산

- 모든 새 페이지는 상태 머신으로
- 기존 페이지는 점진적 리팩토링
- 팀 전체 표준화

## 6️⃣ 반대 의견 대응 전략

### "기존 코드와 다르다"

```
"기존 코드는 기능 중심이었어요.
이 구조는 UX 중심이에요.
QA가 이 구조를 좋아해요."
```

### "복잡해 보인다"

```
"오히려 단순해요.
상태만 보면 UI가 명확해요.
디버깅도 쉬워요."
```

### "시간이 오래 걸린다"

```
"처음만 조금 걸려요.
이후에는 템플릿 복사하면 돼요.
기능 추가도 빨라져요."
```

## 7️⃣ 성공 지표 (측정 가능한 것)

### 정량적 지표

- QA 버그 리포트 감소
- 코드 리뷰 시간 단축
- 새 기능 추가 시간 단축
- 팀원 온보딩 시간 단축

### 정성적 지표

- "이 구조 좋아요" 피드백
- "이해하기 쉬워요" 피드백
- "일관성 있어요" 피드백

---

## 8️⃣ 지속적 개선 전략

### 정기 리뷰

- 월 1회: 상태 머신 구조 리뷰
- 분기 1회: 문서 업데이트
- 반기 1회: 팀 전체 피드백

### 피드백 수집

```
"상태 머신 구조 사용하시면서
불편한 점이나 개선할 점 있으면
언제든 말씀해주세요."
```

이 한 줄이면 대부분 설득됨.
