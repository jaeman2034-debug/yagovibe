# 🔒 최종 아키텍처 정책 (확정본)

## ✅ 최종 결정 요약

**테스트 완료 → 공식 승격 진행 → 테스트 데이터 보관 → 필요 시 관리자 판단으로 삭제**

이 정책은 개발자/행정/감사/실제 운영 모든 관점에서 가장 안전한 선택입니다.

---

## 1️⃣ 공식 승격 후 데이터 상태

| 구분 | 상태 |
|------|------|
| `groups` / `matches` | ✅ 공식 데이터 (활성) |
| `test_groups` / `test_matches` | 🗂 보관 (읽기 전용) |
| 테스트 모드 실행 | ❌ 비활성 |
| 재테스트 | ❌ 불가 (새 대회에서만 가능) |

**👉 운영 중 혼선 0%**

---

## 2️⃣ UX에 반영되는 최종 상태

### OPS > 조 추첨 관리 영역

#### 공식 승격 완료 후
```
🧪 테스트 모드
- 테스트 조 추첨: 완료
- 테스트 대진표: 완료
- 상태: 공식 승격 완료 (테스트 데이터 보관 중)
```

#### 버튼 상태
- ❌ 테스트 조 추첨 실행 (비활성)
- ❌ 테스트 대진표 생성 (비활성)
- ❌ 공식 승격 버튼 (비활성)

#### 하단 관리 옵션 (새로 추가)
```
[테스트 데이터 관리]
테스트 조추첨/대진표 데이터가 보관 중입니다.

[테스트 데이터 삭제] (관리자 전용)
```

---

## 3️⃣ "나중에 삭제" UX 설계

### 버튼 클릭 시 2단계 보호

#### 1차 모달
```
⚠️ 테스트 데이터 삭제

- 테스트 조추첨 및 대진표 데이터가 삭제됩니다.
- 공식 운영 데이터에는 영향이 없습니다.
```

#### 2차 확인 (체크 필수)
```
☑ 테스트 데이터 삭제에 따른 책임을 인지했습니다.
```

**👉 확인 전까지 삭제 불가**

---

## 4️⃣ 삭제 시 시스템 동작 (백엔드)

```typescript
async function deleteTestData(tournamentId) {
  // 1. test_groups 삭제
  await delete(test_groups);
  
  // 2. test_matches 삭제
  await delete(test_matches);
  
  // 3. test_drawLogs 삭제
  await delete(test_drawLogs);
  
  // 4. 감사 로그 기록
  log({
    type: "DELETE_TEST_DATA",
    tournamentId,
    actor: adminId,
    timestamp: now,
  });
}
```

**✔️ 공식 데이터 untouched**
**✔️ 로그는 영구 보관**

## 5️⃣ 왜 "보관 후 삭제"가 최상급 선택이냐

### ✔ 감사 대응
**"사전에 어떤 방식으로 테스트했는가?"**
→ `test_groups` 그대로 증빙 가능

### ✔ 분쟁 대응
**"임의로 조작한 거 아니냐?"**
→ 테스트 → 공식 승격 로그 체인으로 반박

### ✔ 개발 안정성
- 재현 테스트 가능
- 알고리즘 개선 시 비교 가능
