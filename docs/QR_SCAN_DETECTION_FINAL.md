# 🔒 QR 스캔 감지 자동 종료 (FINAL LOCK)

## 최종 선택

**C) 스캔 감지되면 자동 종료**

→ 이것이 현실 + 성공률 기준에서 유일한 정답이다.

---

## 왜 C가 정답인가 (행동 기준 설명)

### QR이 쓰이는 실제 순간

- A가 폰을 들고 있음
- B가 카메라를 켜서 스캔
- 시간: 2–3초
- A의 손/시선은 QR 화면에 묶여 있음

### 이때 필요한 UX는

- ❌ "언제 닫지?" 고민
- ❌ 타이머 인지
- ❌ 버튼 탐색
- ⭕ **아무 생각 없이 보여주기**

👉 **스캔되면 자동 종료 = A의 인지 부담 0**

---

## A/B/C 비교 (냉정한 현실판)

### A) 사용자가 직접 닫기 ❌

- A는 계속 들고 있어야 함
- B가 "됐어" 말해줘야 끝
- 어색한 침묵 발생
- **현장 UX 최악**

### B) 30초 후 자동 종료 ❌

- 대부분 스캔은 3초 안에 끝남
- 30초는 너무 김
- "언제 꺼지지?" 불안
- 타이머는 QR UX에 독

### C) 스캔 감지 시 자동 종료 ⭕

- 스캔 완료 = 목적 달성
- 즉시 이전 화면 복귀
- A의 행동 흐름 유지
- 위챗/알리 계열과 동일

---

## 🔧 구현 정책 (개발자용 LOCK)

### QR 화면 진입

- 화면 밝기 MAX
- 풀스크린
- 다른 UI 요소 없음

### 스캔 감지 이벤트

```javascript
document.addEventListener("visibilitychange", () => {
  if (document.visibilityState === "hidden") {
    closeQrScreen();
  }
});
```

**동작 원리:**
- B가 QR을 스캔하면 새 페이지로 이동
- A의 브라우저 탭/창이 백그라운드로 전환
- `visibilityState`가 "hidden"으로 변경
- 자동으로 QR 화면 종료

👉 **정확도 100% 아니어도 됨**  
👉 **"대충 맞아도 UX는 항상 자연스럽다"**

---

## ❗ 예외 처리 (중요하지만 단순)

### 스캔 실패 시

- 자동 종료 ❌ (visibilitychange는 발생하지 않음)
- A가 원하면:
  - 탭 한 번 → 다시 QR 표시
  - 사용자가 직접 닫기도 가능 (X 버튼 유지)

### 수동 닫기 방법 유지

- 좌상단 X 버튼
- 바깥 영역 탭
- ESC 키
- 스와이프 다운

**이유:** 스캔 실패나 예외 상황 대응

---

## 🔒 최종 UX 흐름 요약

1. 허브 헤더 QR 아이콘 탭
2. → 풀스크린 QR 표시
3. → B가 스캔
4. → 스캔 감지 (visibilitychange)
5. → 자동으로 닫힘
6. → A는 다시 허브 화면

**말 한마디 필요 없음.**

---

## 🧠 천재 모드 핵심 문장 (마지막)

**QR UX의 성공 조건은 "보여준 사람(A)이 불편하지 않은가"다.**

- B의 UX는 이미 완벽하다.
- 이제 A의 UX까지 완성했다.
