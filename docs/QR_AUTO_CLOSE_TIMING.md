# 🔒 QR 자동 종료 타이밍 설계 (FINAL)

## 선택지 분석

### A) 사용자가 닫을 때까지
**장점:**
- 제어권이 사용자에게 있음
- 스캔 실패 시 재시도 가능

**단점:**
- 스캔 성공 후에도 화면 유지 (불편)
- A가 수동으로 닫아야 함
- "스캔 완료했어요" 신호 기다려야 함

---

### B) 30초 후 자동 종료
**장점:**
- 자동으로 닫힘
- 사용자 개입 최소화

**단점:**
- 30초는 너무 김 (스캔은 2-3초면 충분)
- 불필요한 대기 시간
- 스캔 성공 후에도 27초 남음 (낭비)
- A가 화면을 계속 들고 있어야 함

---

### C) 스캔 감지되면 자동 종료 (기술적 불가능)
**장점:**
- 이상적 (스캔 완료 즉시 종료)
- 사용자 경험 최적

**단점:**
- ❌ Web API 제한 (브라우저에서 다른 기기의 스캔 감지 불가)
- ❌ QR 스캔 성공 여부를 A의 브라우저가 알 수 없음
- ❌ 구현 불가능

---

## 🧠 천재 모드 선택: **A + 최소 시간 안내**

### 최종 결정

**A) 사용자가 닫을 때까지 + 시각적 안내**

**이유:**
1. **기술적 제약**: C는 불가능
2. **사용자 제어**: A가 스캔 완료를 확인하고 닫을 수 있음
3. **성공률**: B의 30초는 불필요한 대기 시간 (스캔은 2-3초)
4. **실용성**: A가 "스캔 완료" 신호를 받으면 즉시 닫기 가능

---

## 구현 방향

### 현재 구현 (유지)
- 사용자가 직접 닫기 (X 버튼, 바깥 탭, ESC, 스와이프 다운)
- 자동 종료 없음

### 추가 개선 (선택적)

**시각적 안내 추가:**
```
[ QR 이미지 ]

이 화면을 바로 보여줄 수 있어요

(아주 작게, QR 아래)
스캔 완료 후 화면을 닫아주세요
```

**또는 타이머 표시 (비권장):**
- 5-10초 후에만 "스캔 완료하셨다면 닫아주세요" 메시지 표시
- 자동 종료는 하지 않음

---

## 왜 B (30초 자동 종료)가 아닌가?

### 문제점

1. **불필요한 대기**
   - 스캔은 2-3초면 충분
   - 성공 후 27초는 낭비

2. **사용자 경험 저하**
   - A가 30초 동안 화면을 들고 있어야 함
   - "왜 안 닫히지?" 혼란

3. **성공률과 무관**
   - 30초 타이머는 스캔 성공률과 관계없음
   - 오히려 불편함만 증가

---

## 왜 C (스캔 감지)가 불가능한가?

### 기술적 제약

1. **브라우저 제한**
   - A의 브라우저가 B의 카메라 스캔 결과를 알 수 없음
   - Cross-device 통신 불가능

2. **Web API 한계**
   - Page Visibility API: A의 화면만 감지 가능
   - Network API: QR 스캔은 네트워크 요청 없음
   - Storage API: B의 스캔 결과를 A에게 전달 불가

3. **보안 제약**
   - 브라우저가 다른 기기의 활동을 감지하는 것은 보안 위험
   - Web API로 구현 불가능

---

## 최종 UX 동작 (LOCK)

### 허브 헤더 QR 아이콘 탭

1. 탭
2. → 화면 전환 (풀스크린)
3. → 전체 화면 QR 표시
4. → 배경 흰색
5. → 자동 밝기 MAX

### 닫기 방법

- 좌상단 X 버튼
- 바깥 영역 탭
- ESC 키
- 스와이프 다운

### 자동 종료

- ❌ 자동 종료 없음
- ⭕ 사용자가 직접 닫기

---

## 🔥 핵심 원칙 (최종)

**QR은 "카메라를 위한 UI"다.**

**스캔 성공률 최우선:**
- 풀스크린 + 자동 밝기 → ✅
- 자동 종료 타이머 → ❌ (불필요)

**사용자 제어 유지:**
- 스캔 완료 후 A가 즉시 닫기 가능
- 실패 시 재시도 가능

---
