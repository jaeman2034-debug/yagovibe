# 🧠 상태 머신 설계를 팀에 전파하는 법 (실전용)

> 목표는 단 하나다:
> **"왜 이렇게 만들었냐"는 질문이 안 나오게 만드는 것**

---

## 1️⃣ 문서의 첫 장은 절대 이렇게 쓰지 마라

### ❌ 실패하는 문서

```
- 기술 스택
- 라이브러리
- 컴포넌트 구조
```

**QA / 기획 / 디자이너 전부 안 읽음**

---

## 2️⃣ 정답 문서 첫 장 (이거 그대로 써라)

### 📄 문서 제목

```
[지도 페이지 UI 상태 설계]
– 사용자가 '지금 뭘 하는지' 알게 하기 위함
```

### 📌 첫 문단

> 이 페이지의 목적은
> **길을 안내하기 전에, 상태를 명확히 보여주는 것**이다.
>
> 기능이 없어도 UI는 항상 일관된 상태를 유지한다.

👉 이 문단 하나로 **방향성 고정**

### 📋 문서 구조 예시

```markdown
# [지도 페이지 UI 상태 설계]
## – 사용자가 '지금 뭘 하는지' 알게 하기 위함

### 목적
이 페이지의 목적은 길을 안내하기 전에, 상태를 명확히 보여주는 것이다.
기능이 없어도 UI는 항상 일관된 상태를 유지한다.

### 상태 정의
[검색 중] → [목적지 선택] → [출발 준비] → [안내 중]

각 상태는 완전히 다른 화면 세트를 가진다.

### 구현 원칙
- UI 상태와 기능 상태 분리
- 상태 전환 시 즉각적인 시각 변화
- 기능 실패해도 UI 상태 유지
```

---

## 3️⃣ 상태 머신 그림은 "디자이너 언어"로

### ❌ 코드 다이어그램

```typescript
enum NavUIState {
  SEARCH,
  SELECTED,
  PRE_NAV,
  NAVIGATING
}
```

### ✅ 사람 언어

```
[검색 중]
   ↓
[목적지 선택]
   ↓
[출발 준비]
   ↓
[안내 중]
```

그리고 바로 아래 한 줄:

> 각 상태는 **완전히 다른 화면 세트**를 가진다.

👉 이 한 줄이 **모든 UI 섞임을 막는다**

### 📊 시각적 다이어그램 예시

```
┌─────────────┐
│  검색 중     │  ← 검색바 + 결과 리스트
└──────┬──────┘
       │ [장소 선택]
       ↓
┌─────────────┐
│ 목적지 선택  │  ← 검색바(고정) + 확인 카드
└──────┬──────┘
       │ [여기에 갈게요]
       ↓
┌─────────────┐
│  출발 준비   │  ← 상태바 + 출발 대기 카드
└──────┬──────┘
       │ [출발]
       ↓
┌─────────────┐
│  안내 중     │  ← 턴바이턴 + 안내 카드
└─────────────┘
```

---

## 4️⃣ 리뷰에서 이 말만 반복해라 (만능 문장)

> **"이건 기능 문제가 아니라, 상태가 섞인 문제예요."**

### 이 말의 효과

- ✅ 프론트 설득됨
- ✅ 디자이너 고개 끄덕
- ✅ 기획 반박 불가
- ✅ QA 납득

### ⚠️ 절대 하지 말 것

❌ "UX가 별로라서요"
→ 주관 싸움 된다

### 사용 예시

#### 코드 리뷰에서

```
리뷰어: "이 부분 왜 이렇게 했어요?"

너: "이건 기능 문제가 아니라, 상태가 섞인 문제예요.
     PRE_NAV 상태인데 SEARCH UI가 보이고 있어요."
```

#### 버그 리포트에서

```
버그: "출발 버튼 눌렀는데 아무 변화 없어요"

너: "이건 기능 문제가 아니라, 상태 전환이 누락된 문제예요.
     PRE_NAV → NAVIGATING 전환 시 UI 교체가 안 되고 있어요."
```

## 5️⃣ PR / 리뷰에서 쓰는 체크 문장 (복붙용)

### PR 템플릿

```markdown
## UI 상태 체크리스트

- [ ] 이 변경이 어떤 UI 상태에 영향을 주는지 명확한가?
- [ ] 다른 상태의 UI가 섞이지 않는가?
- [ ] 상태 전환 시 사용자가 변화를 인지할 수 있는가?
- [ ] 기능 실패해도 UI 상태가 유지되는가?
- [ ] 디버그 배지로 상태 확인 가능한가?
```

### 리뷰 코멘트 템플릿

```
이 변경이 UI 상태 머신 원칙에 맞는지 확인해주세요:

1. 어떤 UI 상태에 영향을 주는가?
2. 다른 상태의 UI가 섞이지 않는가?
3. 상태 전환 시 사용자가 변화를 인지할 수 있는가?

참고: docs/STATE_MACHINE_PRINCIPLES.md
```

👉 이 세 줄로 **리뷰 기준 통일**

---

## 6️⃣ 팀이 반발할 때 쓰는 최종 필살기

### 상황: 반발

> "이거 너무 복잡한데요?"

### 답변 (그대로 써라)

> "복잡한 게 아니라
> **지금까지 상태를 정의 안 해서 복잡해진 거예요**."

### 이 말 이후로

- ✅ 반박 거의 없음
- ✅ 논점 종료
- ✅ 설득 완료

### 추가 설명 (필요 시)

```
"지금까지는 상태가 섞여서 복잡해 보였어요.
상태를 명확히 정의하면 오히려 단순해져요.

예를 들어:
- 지금: '검색 중인데 출발 버튼이 보이는' 복잡함
- 개선: 'SEARCH 상태면 출발 버튼 절대 안 보임' 단순함
```

---

## 7️⃣ 기획자 설득용 한 줄

> **"이 구조면 기능이 늦어져도
> 출시 가능한 화면을 먼저 만들 수 있어요."**

### 👉 기획자는 이 말에 약하다

### 추가 설명 (필요 시)

```
"기능 API가 늦어도:
- UI 상태 머신은 먼저 완성
- QA는 UI/UX 테스트 가능
- 출시 일정 지킬 수 있어요

기능은 나중에 붙이면 되고,
UI는 이미 완성된 상태예요."
```

---

## 8️⃣ QA가 이 구조를 좋아하는 이유 (그대로 전달해라)

### QA 기준

- ✅ 상태별 테스트 가능
- ✅ 재현 쉬움
- ✅ 버그 분류 명확

### 비교

```
❌ "가끔 안 돼요"
✅ "PRE_NAV에서 출발 눌렀을 때 UI 전환 누락"

❌ "버튼이 안 보여요"
✅ "SEARCH 상태인데 출발 버튼이 보이는 버그"

❌ "화면이 이상해요"
✅ "SELECTED 상태인데 검색 결과 리스트가 보이는 버그"
```

### 👉 QA가 너 편 된다

### QA에게 전달할 메시지

```
"이 구조면:
- 버그 리포트가 명확해져요
- 재현이 쉬워져요
- 상태별로 테스트 케이스 작성 가능해요

예: 'PRE_NAV → NAVIGATING 전환 테스트'
```

## 9️⃣ 이 설계를 네 이름으로 남기는 법

### 문서 / 코드 주석 / PR 제목에 반드시 이 문구를 남겨라

```
UI State Machine 기반 설계
```

### 사용 예시

#### 코드 주석

```typescript
/**
 * 🔥 UI State Machine 기반 설계
 * 
 * 이 컴포넌트는 UI 상태 머신 원칙을 따릅니다:
 * - UI 상태와 기능 상태 분리
 * - 상태별 UI 세트 완전 교체
 * - 상태 전환 시 즉각적인 시각 변화
 */
```

#### PR 제목

```
[UI State Machine] 지도 페이지 상태 머신 구조 적용
```

#### 문서 제목

```
[UI State Machine 기반 설계] 지도 페이지 UI 상태 설계
```

### 시간 지나면

- ✅ "이 구조 네가 만들었지?"
- ✅ 자동으로 **오너 인정**
- ✅ 설계 주도권 확보

---

## 🏁 최종 결론 (진짜 중요한 말)

> **천재 설계는 코드가 아니라,
> 팀이 같은 생각을 하게 만드는 구조다**

지금 네가 만든 건:

- 지도 페이지 ❌
- 네비 UI ❌

👉 **팀 전체의 UX 사고 방식**이다.

---

## 🎯 여기까지 오면 끝이다

- ✅ 지도 페이지: **완성**
- ✅ UI 구조: **검증 완료**
- ✅ 팀 설득: **방법 확보**

이제부터 네가 하는 건:

- 기능 얹기
- 유지보수
- 확장

👉 **설계는 다시 흔들리지 않는다.**
