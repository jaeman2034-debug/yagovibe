## Ⅰ. 네 작업 방식의 핵심 구조 (한 줄 요약)

```
문제 인식 → 오해 제거 → UX 기준 고정 → 마감 → 운영 대비
```

**이 순서가 한 번도 깨지지 않았다.**  
**이게 실력이다.**

### 단계별 의미

1. **문제 인식**: 현상이 아닌 느낌을 잡는다
2. **오해 제거**: 원인은 코드보다 구조에서 찾는다
3. **UX 기준 고정**: 구현 전에 판정 기준을 고정한다
4. **마감**: 기능 구현보다 마감을 먼저 끝낸다
5. **운영 대비**: 항상 "운영 중 사고"를 가정한다

## Ⅱ. 5단계 완주 프레임워크

### ① 현상 말고 느낌을 잡는다

#### ❌ 나쁜 접근
- "클릭이 안 된다"
- "버튼이 작동하지 않는다"
- "이벤트가 발생하지 않는다"

#### ⭕ 좋은 접근
- "안 눌린 것처럼 느껴진다"
- "반응이 없는 것 같다"
- "불안한 느낌이 든다"

#### 왜 중요한가?
**기술 문제 이전에 인지 문제를 먼저 본다.**

- 사용자는 기술적 원인을 모른다
- 사용자는 느낌으로 판단한다
- 느낌을 고치면 기술 문제도 자연히 해결된다

#### 실행 방법
1. **사용자 관점에서 질문하기**
   - "사용자가 이 순간 무엇을 느낄까?"
   - "이 느낌이 불안한가? 불편한가?"
   - "이 느낌이 신뢰를 주는가?"

2. **기술 용어 제거**
   - "이벤트 핸들러" → "버튼 반응"
   - "상태 관리" → "화면 변화"
   - "API 호출" → "작업 완료"

3. **감정 단어 사용**
   - 불안, 짜증, 신뢰, 안심, 혼란, 확신

---

### ② 원인은 코드보다 구조에서 찾는다

#### 흔한 오해
- "이벤트가 안 들어온다" → 이벤트 핸들러 문제?
- "버튼이 안 보인다" → CSS 문제?
- "상태가 안 바뀐다" → useState 문제?

#### 실제 원인 (90%는 구조 문제)

##### z-index 충돌
```typescript
// 문제: 메뉴가 모달 뒤에 가려짐
// 원인: z-index 계층 구조 불명확
// 해결: z-index 체계 명확히 정의
const Z_INDEX = {
  menu: 99999,
  modal: 100000,
  toast: 100001,
};
```

##### fixed / sticky 충돌
```typescript
// 문제: 메뉴 위치가 이상함
// 원인: fixed와 sticky가 충돌
// 해결: position 계층 구조 명확히
const menuStyle = {
  position: 'fixed', // absolute가 아닌 fixed
  top: HEADER_HEIGHT + 48,
  right: 12,
};
```

##### 상태 소유권 불명확
```typescript
// 문제: 상태가 예상과 다르게 동작
// 원인: 누가 상태를 소유하는지 불명확
// 해결: 상태 소유권 명확히 정의
// - 메뉴 상태: ChatRoom 컴포넌트
// - 모달 상태: ChatRoom 컴포넌트
// - 전역 상태: Context 또는 Redux
```

#### 실행 방법
1. **구조 다이어그램 그리기**
   - 컴포넌트 계층 구조
   - 상태 흐름도
   - 이벤트 흐름도

2. **레이어링 체계 정의**
   - z-index 체계
   - position 체계
   - 상태 소유권 체계

3. **충돌 지점 찾기**
   - 같은 레이어에 있는 요소들
   - 같은 상태를 사용하는 컴포넌트들
   - 같은 이벤트를 듣는 핸들러들

👉 **"이벤트가 안 들어온다"는 말이 나오면 90%는 구조 문제다.**

---

### ③ UX 합격선을 먼저 정의한다

#### 구현 전에 판정 기준을 고정

##### 질문 1: 언제 닫혀야 하는가?
- 외부 클릭 시
- ESC 키 입력 시
- 라우트 변경 시
- 모달 열림 시

##### 질문 2: 언제 안심해야 하는가?
- 행동 후 즉시 피드백
- 상태가 명확히 표시됨
- 실패해도 복구 가능

##### 질문 3: 언제 짜증이 나는가?
- 반응이 느림 (>50ms)
- 로딩이 너무 길거나 짧음
- 에러 메시지가 불명확함

#### 실행 방법
1. **UX 기준 문서 작성**
   ```markdown
   ## 메뉴 UX 기준
   - 반응 시간: <50ms
   - 닫힘 조건: 외부 클릭, ESC, 라우트 변경
   - 애니메이션: 80ms 딜레이
   - 터치 영역: 최소 44x44px
   ```

2. **판정 질문 리스트**
   - "안 눌린 느낌" 0인가?
   - 행동 결과가 즉시 보이는가?
   - 실수해도 복구 가능한가?

3. **테스트 시나리오 작성**
   - 각 기준을 검증하는 테스트 케이스
   - 합격/불합격 기준 명확히

👉 **구현 전에 판정 기준을 고정한다.**

### ④ 기능 구현보다 마감을 먼저 끝낸다

#### 마감 체크리스트

##### 외부 클릭
- [ ] 메뉴 외부 클릭 시 닫힘
- [ ] 모달 외부 클릭 시 닫힘 (또는 유지)
- [ ] 이벤트 버블링 처리

##### ESC
- [ ] ESC 키로 메뉴 닫힘
- [ ] ESC 키로 모달 닫힘
- [ ] 포커스 트랩 작동

##### 연타
- [ ] 버튼 연타 방지
- [ ] 중복 요청 가드
- [ ] 상태 꼬임 없음

##### 모바일
- [ ] 터치 영역 44px+
- [ ] 스크롤 잠금 작동
- [ ] 키보드 대응
- [ ] 실기기 테스트

#### 실행 방법
1. **마감 우선순위**
   - 기능 구현보다 마감이 먼저
   - "일단 작동하게"보다 "안정적으로 작동하게"

2. **마감 체크리스트 사용**
   - 각 기능마다 마감 체크리스트 적용
   - 체크리스트 완료 전 배포 금지

3. **마감 자동화**
   - 자동화 테스트로 마감 검증
   - CI/CD 파이프라인에 마감 테스트 포함

👉 **이걸 끝내면 기능은 자연히 안정된다.**

### ⑤ 항상 "운영 중 사고"를 가정한다

#### 운영 대비 체크리스트

- [ ] 오프라인 상태 감지

##### 로그
- [ ] 핵심 이벤트 로그
- [ ] 에러 로그 (컨텍스트 포함)
- [ ] 성능 로그

##### 롤백
- [ ] 기능 플래그
- [ ] 이전 빌드 보관
- [ ] 롤백 절차 문서화

##### 가드
- [ ] 서버 가드
- [ ] 클라이언트 가드
- [ ] 이중 방어

#### 실행 방법
1. **사고 시나리오 작성**
   - "만약 이게 실패하면?"
   - "사용자가 이상한 짓을 하면?"
   - "서버가 응답하지 않으면?"

2. **대응 계획 수립**
   - 각 시나리오별 대응 방법
   - 롤백 계획
   - 사용자 공지 템플릿

3. **모니터링 설정**
   - 에러 알림
   - 성능 모니터링
   - 사용자 행동 추적

👉 **이 단계가 있으면 망하지는 않는다.**

- 즉시 적용 가능
