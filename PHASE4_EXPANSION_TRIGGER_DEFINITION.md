# Phase 4 확장 트리거 정의

**목적**: 언제, 어떤 신호가 오면 "자동화(정산/기록)"로 넘어가는지 조건만 정한다 (코드 ❌, 기준만 ⭕)

---

## 1️⃣ Phase 4 확장의 의미

### Phase 4 확장 = 자동화(정산/기록) 단계

**핵심 질문:**
**"Phase 3-4-1의 '보여주고 잠그는 것'에서, 언제 '자동화(정산/기록)'로 넘어가야 하는가?"**

---

## 2️⃣ Phase 4로 넘어가는 명확한 조건 (3개 중 2개 이상)

### ✅ 조건 1: 직접 수정 요구 발생

#### 정의
**사무국/운영자가 말한다: "이건 내가 직접 고쳐야겠다."**

#### 의미
- Edit Mode 실사용 필요성 발생
- 공개 정보만으로는 부족, 실제 수정 기능 필요

#### 판정 기준
**사무국/운영자가 명확하게 "직접 수정" 요구**

---

### ✅ 조건 2: 동일 수기 작업 주 3회 이상

#### 정의
**동일한 수기 작업이 주 3회 이상 반복**

#### 예시
- 대관 상태 변경을 반복 메모
- 공지/일정 수정 요청 누적
- 스프레드시트로 대관 시간표 수정 (주 3회 이상)

#### 의미
- 잠금 → 기록 → 자동화로 넘어갈 신호
- 반복 작업이 명확하게 발생

#### 판정 기준
**동일 수기 작업이 주 3회 이상 반복 (2주 이상 지속)**

---

### ✅ 조건 3: 문의 감소가 '한계'에 도달

#### 정의
**Phase 3로 문의는 줄었으나, 더 줄이려면 입력/수정 자동화가 필요해짐**

#### 의미
- 정보 제공만으로는 부족
- 실제 자동화 시스템 필요

#### 판정 기준
**Phase 3 운영 후 문의 감소가 정체되었으나, 추가 감소를 위해서는 자동화 필요**

---

### Phase 4 확장 판정 기준

**다음 3개 조건 중 2개 이상 충족 시 → Phase 4 확장 착수**

| 조건 | 판정 기준 |
|------|----------|
| 직접 수정 요구 발생 | 명확한 요구 |
| 동일 수기 작업 주 3회 이상 | 2주 이상 지속 |
| 문의 감소가 '한계'에 도달 | 정체 + 추가 감소 필요 |

---

## 3️⃣ ❌ Phase 4로 안 넘어가는 경우

### 제외 사항
1. **단순 불편함**
   - 일시적인 불편함
   - 해결 방법이 있는 불편함

2. **"있으면 좋겠다" 수준의 아이디어**
   - 명확한 필요성 없음
   - 단순 기능 요청

3. **한 번짜리 이슈**
   - 반복되지 않는 이슈
   - 일회성 문제

### 핵심 원칙
**반복 + 요구 + 시간 절감이 동시에 있어야 한다**

---

## 4️⃣ Phase 4 확장 범위 (확장 시 구현할 기능)

### Phase 4-2: FacilitySection 자동화
- 관리자가 슬롯 상태 토글 (`available` ↔ `blocked`)
- 상태 변경 시 자동 알림 발송 (Kakao Alimtalk)
- 전화 문의 0건 달성

### Phase 4-3: TournamentSection 연동
- Tournament 데이터와 Hero Section 자동 연동
- Tournament → FacilitySection 자동 연결 (event 슬롯)
- 관리자가 Tournament만 수정하면 모든 화면 자동 업데이트

### Phase 4-4: 협회 산하 팀 표시
- 협회 공식 페이지에 산하 팀 목록 표시 (읽기 전용)
- 축구 허브 팀 카드에 배지 표시 (회원/연계)
- "우리 팀은 산하인가요?" 전화 문의 감소

---

## 5️⃣ 확장하지 않는 것 (Phase 4에서도 ❌)

### Phase 4에서도 하지 않는 것
- ❌ 예약/신청 시스템 (Phase 5 이후)
- ❌ 결제 시스템 (Phase 5 이후)
- ❌ 승인 워크플로우 (Phase 5 이후)
- ❌ 엑셀 업로드 (Phase 5 이후)

**이유:** Phase 4는 "자동화(정산/기록)", Phase 5는 "승인/관리 시스템"

---

## 6️⃣ 확장 트리거 모니터링 방법

### 일일 체크 (운영자)
- 전화/KakaoTalk 문의 횟수 기록
- 수기 작업 빈도 기록
- 시간 소모 기록

### 주간 체크 (운영자)
- 동일 질문 반복 패턴 확인
- 수기 작업 반복 패턴 확인

### 월간 체크 (시스템)
- 페이지 조회수 확인
- 데이터 축적 정도 확인
- 에러 발생 횟수 확인

---

## 7️⃣ 확장 트리거 판정 프로세스

### 1단계: 데이터 수집 (2주 이상)
- 운영자: 문의/작업 빈도 기록
- 시스템: 페이지 접근 통계 수집
- 사용자: 피드백 수집

### 2단계: 트리거 확인 (주간)
- 4개 트리거 중 충족 개수 확인
- 3개 이상 충족 시 다음 단계 진행

### 3단계: 확장 검토
- Phase 4 확장 범위 확인
- 우선순위 결정 (4-2 → 4-3 → 4-4)
- 구현 계획 수립

### 4단계: 확장 착수
- Phase 4 확장 기능 구현 시작
- Phase 3-4-1 운영은 계속 유지
