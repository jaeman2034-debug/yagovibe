# "지금은 없는 기능" 명시적 선언

**목적**: 안 해서 더 강해지는 설계 - 왜 지금 이 기능이 없는지 명시적으로 선언하여 기획이 흔들리지 않게 함

---

## 1️⃣ 핵심 원칙

### 원칙
**"지금은 없는 기능"을 명시적으로 선언하면 기획이 흔들리지 않는다**

### 목적
- 기능 요청 시 "지금은 안 하는 것"으로 명확히 거절 가능
- 설계 의도를 문서화하여 향후 판단 기준 확립
- "안 해서 더 강해지는" 설계 원칙 명확화

---

## 2️⃣ 공식 선언문 (협회 기준선)

### ❌ 지금 하지 않는 기능 (의도적 제외)

1. **팀 생성 / 가입 신청**
2. **협회 페이지 내 문의 버튼**
3. **대관 신청 / 승인 / 결제**
4. **Edit Mode 일반 공개**
5. **일정·대회 엑셀 업로드**

### 이유
**공식 기준을 먼저 세우고, 자동화는 나중에 한다.**
**지금은 "보여주고 잠그는 것"이 최우선.**

---

## 3️⃣ 협회 페이지에서 하지 않는 것 (명시적 선언 - 상세)

### ❌ 팀 생성

#### 선언
**협회 공식 페이지에서 팀 생성 기능은 제공하지 않는다**

#### 이유
1. **역할 분리**: 협회 페이지는 "정보 확인" 장소, 팀 생성은 "참여 활동" 장소
2. **흐름 정리**: 팀 생성은 축구 허브 → "팀 / FC 찾기"에서 처리
3. **IA 일관성**: 협회(공식) → 팀(참여) → 개인(활동) 흐름 유지

#### 언제 할 것인가?
- Phase 4 이후 검토 (협회 제안 기능 포함 시)

---

### ❌ 가입 신청

#### 선언
**협회 공식 페이지에서 협회 산하 가입 신청 기능은 제공하지 않는다**

#### 이유
1. **Phase 3-4-1 범위**: 승인 워크플로우는 Phase 5 이후
2. **역할 혼동 방지**: 협회 페이지는 정보 확인, 신청은 별도 플로우
3. **분쟁 방지**: 신청 기능이 있으면 "왜 승인 안 되냐" 문의 증가

#### 언제 할 것인가?
- Phase 5: 승인 워크플로우 구현 시

---

### ❌ 문의 버튼

#### 선언
**협회 공식 페이지에 문의 버튼/링크를 제공하지 않는다**

#### 이유
1. **전화 문의 차단 목적**: 문의 버튼이 있으면 전화 문의가 증가함
2. **정보 자급**: 모든 정보가 페이지에 있으므로 문의 불필요
3. **운영 효율**: 사무국장의 전화 부담 증가 방지

#### 언제 할 것인가?
- Phase 4 이후: 자동화 시스템이 완성된 후 검토

---

### ❌ 승인 플로우

#### 선언
**협회 공식 페이지에서 승인 워크플로우는 제공하지 않는다**

#### 이유
1. **Phase 5 기능**: 승인 워크플로우는 Phase 5에서 구현
2. **역할 분리**: 공식 페이지는 Public 정보 확인, 승인은 Admin 기능
3. **복잡도 관리**: Phase 3-4-1은 정보 확인 중심으로 단순 유지

#### 언제 할 것인가?
- Phase 5: 승인/관리 시스템 구현 시

---

## 3️⃣ Hero Section에서 하지 않는 것

### ❌ 버튼/CTA

#### 선언
**Hero Section에 버튼, CTA, 클릭 유도 요소를 제공하지 않는다**

#### 이유
1. **역할 명확**: Hero는 "정의하는 곳", 행동 유도 아님
2. **단순성 유지**: 복잡한 UI는 인식 방해
3. **권위 전달**: 버튼 없음 = 행동 요구 없음 = 기준 선언만

#### 언제 할 것인가?
- Phase 4 이후: 필요 시 Quick Links 추가 검토 (현재는 불필요)

---

### ❌ 애니메이션

#### 선언
**Hero Section에 애니메이션 효과를 추가하지 않는다**

#### 이유
1. **공식성 강조**: 애니메이션은 홍보성, Hero는 공식 선언
2. **인식 속도**: 3초 안에 인지해야 하므로 방해 요소 제거
3. **접근성**: 애니메이션은 일부 사용자에게 방해 요소

#### 언제 할 것인가?
- Phase 4 이후: 사용자 피드백 기반 검토 (현재는 불필요)

---

## 4️⃣ 축구 허브에서 하지 않는 것

### ❌ 협회 페이지에서 팀으로 직접 이동

#### 선언
**축구 허브 공식 카드 클릭 시 팀 목록으로 직접 이동하지 않는다**

#### 이유
1. **흐름 정리**: 협회 → (허브) → 팀 (간접 흐름)
2. **역할 분리**: 협회 카드는 협회 페이지로만 이동
3. **IA 일관성**: 공식 주체 → 참여 주체 순서 유지

#### 언제 할 것인가?
- Phase 4 이후: 사용자 피드백 기반 검토 (현재는 불필요)

---

## 5️⃣ 팀 페이지에서 하지 않는 것 (Phase 3-4-1)

### ❌ 협회 관계 수정

#### 선언
**팀 페이지에서 협회 관계(산하/연계)를 수정할 수 없다**

#### 이유
1. **협회 권한**: 관계는 협회가 관리하는 것
2. **데이터 무결성**: 팀이 임의로 관계 변경 시 데이터 혼란
3. **Phase 5 기능**: 관계 변경은 Phase 5 승인 워크플로우로만

#### 언제 할 것인가?
- Phase 5: 승인 워크플로우 구현 시 (신청만 가능)

---

## 6️⃣ 전체적으로 하지 않는 것 (Phase 3-4-1)

### ❌ 승인 워크플로우

#### 선언
**Phase 3-4-1에서는 승인 워크플로우(신청/승인/거절)를 구현하지 않는다**

#### 이유
1. **Phase 5 기능**: 승인 워크플로우는 Phase 5에서 구현
2. **복잡도 관리**: Phase 3-4-1은 정보 확인 중심
3. **단계적 확장**: 기본 정보 구조 확립 후 승인 시스템 추가

#### 언제 할 것인가?
- Phase 5: 승인/관리 시스템 구현 시

---

### ❌ 엑셀 업로드/일괄 등록

#### 선언
**대회/팀/일정 등의 엑셀 업로드 또는 일괄 등록 기능을 제공하지 않는다**

#### 이유
1. **데이터 품질**: 수동 입력이 데이터 품질 보장
2. **오류 방지**: 엑셀 업로드는 오류 발생 가능성 높음
3. **복잡도 관리**: Phase 3-4-1은 수동 입력 중심

#### 언제 할 것인가?
- Phase 4 이후: 운영 안정화 후 검토

---

## 8️⃣ 지금 하는 기능 (Phase 3–4 고정)

### ✅ 현재 구현/제공하는 기능

1. **공식 대회·일정 열람**
   - 협회 공식 페이지에서 대회 정보 확인
   - TournamentSection 읽기 전용

2. **대관 상태 확인(잠금/이벤트)**
   - FacilitySection에서 슬롯 상태 확인
   - available / blocked / event 상태 표시

3. **협회 공지·스토리 공식 공표**
   - NoticeSection, StorySection 읽기 전용
   - 공식 정보 확인

4. **팀/개인은 참조만 가능**
   - 산하 팀 목록 읽기 전용
   - 팀 페이지로 이동 (참조)

### 핵심 원칙
**지금은 "보여주고 잠그는 것"이 최우선**
- 정보 확인 중심
- 행동 유도 최소화
- 공식 기준 선언

## 9️⃣ 분쟁 방지 문구 (페이지 공통)

### 공식 문구 (모든 페이지 하단 또는 적절한 위치)
```
본 페이지의 정보는 노원구 축구협회가 관리하는 공식 기준입니다.
```

### 효과
1. **회원 vs 비회원 분쟁 차단**
   - 공식 기준임을 명시
   - 개인 판단/해석 여지 제거

2. **"말 바뀜" 방지**
   - 공식 문구로 고정
   - 사무국 개인 책임으로 오해 방지

3. **사무국 개인 책임 제거**
   - 협회가 관리하는 공식 기준임을 명시
   - 개인 판단/해석이 아닌 공식 기준임을 강조

### 표시 위치
- 협회 공식 페이지 하단 (권장)
- 또는 각 섹션 하단

## 🔟 "안 해서 더 강해지는" 설계 원칙

### 원칙 1: 단순함이 신뢰를 만든다
- 복잡한 기능보다 단순한 정보 제공이 공식성 강조
- 버튼/CTA 없음 = 행동 요구 없음 = 기준 선언만

### 원칙 2: 역할 분리가 명확성을 만든다
- 협회 페이지 = 정보 확인
- 축구 허브 = 활동 진입점
- 팀 페이지 = 팀 관리

### 원칙 3: 단계적 확장이 안정성을 만든다
- Phase 3-4-1: 정보 확인 중심
- Phase 4: 관계 표시
- Phase 5: 승인 워크플로우
