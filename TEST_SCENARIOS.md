# 🧪 채팅 안전감 강화 기능 - 테스트 시나리오

> **목표**: 사람이 놓치는 케이스까지 커버하여 출시 후 "이상한 버그" 확률 최소화

---

## 1️⃣ 점 3개 메뉴 UX 테스트

### 기본 동작

#### TC-001: 메뉴 열기/닫기
- **전제조건**: 채팅방 진입 완료
- **실행**:
  1. 헤더 우측 상단 점 3개 버튼 클릭
- **기대결과**: 
  - 메뉴가 우측 상단에서 부드럽게 열림 (scale 애니메이션)
  - 메뉴 내부에 "신고", "차단" 옵션 표시
  - 배경 오버레이 표시

#### TC-002: 메뉴 닫기 (다시 클릭)
- **전제조건**: 메뉴가 열린 상태
- **실행**:
  1. 점 3개 버튼 다시 클릭
- **기대결과**: 
  - 메뉴가 부드럽게 닫힘 (80ms 딜레이)
  - 배경 오버레이 제거

#### TC-003: 외부 클릭으로 닫기
- **전제조건**: 메뉴가 열린 상태
- **실행**:
  1. 메뉴 외부 영역 클릭
- **기대결과**: 
  - 메뉴가 즉시 닫힘
  - 스크롤 잠금 해제

#### TC-004: ESC 키로 닫기
- **전제조건**: 메뉴가 열린 상태
- **실행**:
  1. ESC 키 누름
- **기대결과**: 
  - 메뉴가 즉시 닫힘
  - 포커스가 메뉴 버튼으로 복귀

### 경계 케이스

#### TC-005: 연타 테스트
- **전제조건**: 채팅방 진입 완료
- **실행**:
  1. 점 3개 버튼을 5회 연속 빠르게 클릭
- **기대결과**: 
  - 메뉴 상태가 꼬이지 않음
  - 최종적으로 열림 또는 닫힘 상태 중 하나로 안정화
  - 콘솔 에러 없음

#### TC-006: 라우트 이동 시 메뉴 닫힘
- **전제조건**: 메뉴가 열린 상태
- **실행**:
  1. 뒤로가기 버튼 클릭하여 채팅 목록으로 이동
- **기대결과**: 
  - 메뉴가 자동으로 닫힘
  - 메모리 누수 없음 (useEffect cleanup 작동)

#### TC-007: 스크롤 중 메뉴 열기
- **전제조건**: 채팅 메시지가 많아 스크롤 가능한 상태
- **실행**:
  1. 채팅을 중간 지점으로 스크롤
  2. 점 3개 버튼 클릭
- **기대결과**: 
  - 메뉴가 고정 위치(우측 상단)에 정확히 표시
  - 스크롤 위치가 변경되지 않음
  - 스크롤 잠금 작동

### 모바일 테스트

#### TC-008: 한 손 터치 가능
- **전제조건**: 모바일 디바이스 (iPhone SE 크기)
- **실행**:
  1. 엄지로 점 3개 버튼 터치
- **기대결과**: 
  - 버튼 터치 영역이 최소 44x44px
  - 터치 미스 없음
  - 메뉴가 정확히 열림

#### TC-009: 스크롤 잠금 작동
- **전제조건**: 모바일, 메뉴가 열린 상태
- **실행**:
  1. 메뉴 열린 상태에서 화면을 스와이프하여 스크롤 시도
- **기대결과**: 
  - 배경 스크롤이 잠김 (`document.body.style.overflow = 'hidden'`)
  - 메뉴 내부는 스크롤 가능 (필요시)

#### TC-010: 터치 미스 없음
- **전제조건**: 모바일 디바이스
- **실행**:
  1. 메뉴 옵션("신고", "차단")을 빠르게 연속 터치
- **기대결과**: 
  - 의도한 옵션만 선택됨
  - 중복 선택 없음
  - 터치 피드백 즉각 반응 (< 50ms)

## 2️⃣ 신고 플로우 테스트

### 정상 플로우

#### TC-011: 신고 확인 모달 표시
- **전제조건**: 메뉴가 열린 상태
- **실행**:
  1. "신고" 옵션 클릭
- **기대결과**: 
  - 메뉴가 닫힘
  - 신고 확인 모달이 표시됨
  - 모달 내용: "신고하시겠습니까?", "이 사용자의 채팅 내용이 검토됩니다.", "신고 접수 후 검토가 진행되며, 필요시 조치가 취해집니다."

#### TC-012: 신고 취소
- **전제조건**: 신고 확인 모달이 열린 상태
- **실행**:
  1. "취소" 버튼 클릭
- **기대결과**: 
  - 모달이 닫힘
  - 아무 변경사항 없음
  - 채팅 상태 정상 유지

#### TC-013: 신고 확인 (성공)
- **전제조건**: 신고 확인 모달이 열린 상태, 네트워크 정상
- **실행**:
  1. "신고" 버튼 클릭
- **기대결과**: 
  - 버튼이 "처리 중..."으로 변경 (300ms 후)
  - 신고가 Firestore에 저장됨 (`chatReports` 컬렉션)
  - 사용자 위험도 업데이트 (`users` 컬렉션의 `reportCount`, `riskLevel`)
  - 토스트 메시지: "필요한 조치를 검토 중입니다." (4000ms 표시)
  - 채팅에 시스템 메시지 추가: "⚠️ 신고가 접수되었고, 현재 검토 중입니다."
  - 모달이 닫힘

### 중복 신고 방지

#### TC-014: 중복 신고 차단
- **전제조건**: 이미 신고가 접수된 채팅방
- **실행**:
  1. "신고" 옵션 클릭
  2. "신고" 버튼 클릭
- **기대결과**: 
  - 모달이 닫힘
  - 토스트 메시지: "이미 신고가 접수되었습니다." (3000ms)
  - 서버 요청이 발생하지 않음 (중복 체크로 차단)

#### TC-015: 새로고침 후 상태 유지
- **전제조건**: 신고가 접수된 채팅방
- **실행**:
  1. 채팅방 새로고침 (F5 또는 페이지 리로드)
  2. 다시 "신고" 시도
- **기대결과**: 
  - 서버에서 중복 신고 상태 확인
  - "이미 신고가 접수되었습니다." 메시지 표시
  - 시스템 메시지가 채팅 내에 유지됨

### 실패 케이스

#### TC-016: 네트워크 오류 처리
- **전제조건**: 네트워크 연결 끊김 또는 서버 오류
- **실행**:
  1. "신고" 옵션 클릭
  2. "신고" 버튼 클릭
- **기대결과**: 
  - 에러 토스트: "요청에 실패했습니다. 잠시 후 다시 시도해주세요." (3000ms)
  - 모달이 닫힘
  - 로딩 스피너가 사라짐
  - 재시도 가능 (모달을 다시 열어 신고 재시도)

#### TC-017: 신고 연타 방지
- **전제조건**: 신고 확인 모달이 열린 상태
- **실행**:
  1. "신고" 버튼을 5회 연속 빠르게 클릭
- **기대결과**: 
  - 서버 요청이 1회만 발생
  - `isReporting` 상태로 중복 요청 차단
  - 버튼이 "처리 중..." 상태로 비활성화

## 3️⃣ 차단 플로우 테스트

### 정상 플로우

#### TC-018: 차단 모달 표시
- **전제조건**: 메뉴가 열린 상태
- **실행**:
  1. "차단" 옵션 클릭
- **기대결과**: 
  - 메뉴가 닫힘
  - 차단 확인 모달이 표시됨
  - 차단 이유 선택 옵션 표시: "불쾌한 언행", "스팸 / 광고", "원치 않는 연락", "기타"
  - 버튼 텍스트: "이유 없이 차단" (이유 미선택 시)

#### TC-019: 이유 선택 후 차단
- **전제조건**: 차단 모달이 열린 상태
- **실행**:
  1. "불쾌한 언행" 라디오 버튼 선택
  2. "차단" 버튼 클릭
- **기대결과**: 
  - 버튼 텍스트가 "차단"으로 변경됨
  - 버튼이 "처리 중..."으로 변경 (300ms 후)
  - Firestore에 차단 기록 저장 (`users/{uid}/blockedUsers/{otherUid}`)
  - `reason: "abuse"` 필드 저장됨
  - 채팅 목록으로 이동 (`/app/chat`)
  - 토스트 메시지: "차단되었습니다." (3000ms)

#### TC-020: 이유 없이 차단
- **전제조건**: 차단 모달이 열린 상태
- **실행**:
  1. 이유 선택 없이 "이유 없이 차단" 버튼 클릭
- **기대결과**: 
  - 차단이 정상 처리됨
  - Firestore에 `reason` 필드 없이 저장됨 (또는 `reason: undefined`)
  - 채팅 목록으로 이동
  - 토스트 메시지: "차단되었습니다."

### 재진입 차단

#### TC-021: 뒤로가기로 재진입 차단
- **전제조건**: 사용자가 차단한 후 채팅 목록으로 이동
- **실행**:
  1. 브라우저 뒤로가기 버튼 클릭
- **기대결과**: 
  - 채팅방에 접근 불가
  - 자동으로 채팅 목록으로 리다이렉트 (`navigate("/app/chat", { replace: true })`)
  - 또는 차단 안내 메시지 표시

#### TC-022: URL 직접 접근 차단
- **전제조건**: 사용자가 차단한 사용자
- **실행**:
  1. 브라우저 주소창에 채팅방 URL 직접 입력
  2. Enter 키 입력
- **기대결과**: 
  - 채팅방 진입 시도 시 차단 상태 확인
  - 자동으로 채팅 목록으로 리다이렉트
  - 또는 "차단된 사용자입니다" 메시지 표시

#### TC-023: 새로고침 후 재진입 차단
- **전제조건**: 차단된 사용자의 채팅방 URL에 있음
- **실행**:
  1. 페이지 새로고침 (F5)
- **기대결과**: 
  - 서버에서 차단 상태 확인
  - 자동으로 채팅 목록으로 리다이렉트
  - 차단 상태가 유지됨

### 실패 케이스

#### TC-024: 서버 실패 처리
- **전제조건**: 네트워크 연결 끊김 또는 서버 오류
- **실행**:
  1. "차단" 옵션 클릭
  2. 이유 선택 (또는 선택 안 함)
  3. "차단" 버튼 클릭
- **기대결과**: 
  - 에러 토스트: "요청에 실패했습니다. 잠시 후 다시 시도해주세요." (3000ms)
  - 모달이 닫히지 않음 (재시도 가능하도록)
  - 로딩 스피너가 사라짐
  - 재시도 가능

#### TC-025: 차단 연타 방지
- **전제조건**: 차단 모달이 열린 상태
- **실행**:
  1. "차단" 버튼을 5회 연속 빠르게 클릭
- **기대결과**: 
  - 서버 요청이 1회만 발생
  - `isBlocking` 상태로 중복 요청 차단
  - 버튼이 "처리 중..." 상태로 비활성화

## 5️⃣ 시각적 회귀 테스트

### 텍스트 오버플로우

#### TC-031: 긴 닉네임 처리
- **전제조건**: 상대방 닉네임이 매우 긴 경우 (예: "이것은매우긴닉네임입니다123456789")
- **실행**:
  1. 채팅방 진입
- **기대결과**: 
  - 헤더에 닉네임이 `truncate` 처리됨
  - `max-width: 180px` 적용
  - `text-overflow: ellipsis` 표시
  - 레이아웃이 깨지지 않음
  - 점 3개 버튼이 밀리지 않음

#### TC-032: 다국어 텍스트
- **전제조건**: 닉네임이 한글/영어/이모지 혼합 (예: "안녕Hello👋123")
- **실행**:
  1. 채팅방 진입
- **기대결과**: 
  - 모든 문자가 정상 표시
  - 이모지가 깨지지 않음
  - 레이아웃이 안정적

#### TC-033: 이모지만 있는 닉네임
- **전제조건**: 닉네임이 이모지만 (예: "👋😊🎉")
- **실행**:
  1. 채팅방 진입
- **기대결과**: 
  - 이모지가 정상 표시
  - 레이아웃이 안정적

### 화면 크기

#### TC-034: 작은 화면 (iPhone SE)
- **전제조건**: iPhone SE 크기 (375x667)
- **실행**:
  1. 채팅방 진입
  2. 메뉴 열기
  3. 모달 열기
- **기대결과**: 
  - 모든 요소가 화면에 맞게 표시
  - 텍스트가 잘리지 않음
  - 버튼 터치 영역이 최소 44x44px
  - 스크롤 가능한 영역이 정상 작동

#### TC-035: 큰 화면 (데스크탑)
- **전제조건**: 데스크탑 화면 (1920x1080)
- **실행**:
  1. 채팅방 진입
  2. 메뉴 열기
- **기대결과**: 
  - 레이아웃이 중앙 정렬
  - 메뉴가 적절한 위치에 표시
  - 호버 효과 정상 작동

### 다크모드

#### TC-036: 다크모드 색상
- **전제조건**: 다크모드 활성화
- **실행**:
  1. 채팅방 진입
  2. 메뉴 열기
  3. 모달 열기
- **기대결과**: 
  - 모든 텍스트가 가독성 있게 표시
  - 배경색이 다크 테마에 맞음
  - 버튼 색상이 적절히 대비됨
  - 그림자가 다크모드에 맞게 조정됨 (`--shadow-sm`, `--shadow-md`)

#### TC-037: 다크모드 경계선
- **전제조건**: 다크모드 활성화
- **실행**:

- **기대결과**:

  - 요소 구분이 명확함

## 6️⃣ 퍼포먼스 체감 테스트

#### TC-038: 메뉴 반응 속도
- **전제조건**: 채팅방 진입 완료
- **실행**:
  1. 점 3개 버튼 클릭
  2. 클릭 시점부터 메뉴가 보일 때까지 시간 측정
- **기대결과**: 
  - 반응 시간 < 50ms
  - 애니메이션이 부드러움 (60fps)
  - 버벅임 없음

#### TC-039: API 응답 속도 (빠른 경우)
- **전제조건**: 네트워크가 빠름 (< 300ms 응답)
- **실행**:
  1. "신고" 또는 "차단" 버튼 클릭
- **기대결과**: 
  - 로딩 스피너가 표시되지 않음 (300ms 이내 완료)
  - 즉각적인 피드백

#### TC-040: API 응답 속도 (느린 경우)
- **전제조건**: 네트워크가 느림 (> 300ms 응답)
- **실행**:
  1. "신고" 또는 "차단" 버튼 클릭
- **기대결과**: 
  - 300ms 후 로딩 스피너 표시
  - "처리 중..." 텍스트 표시
  - 사용자가 진행 상황을 인지할 수 있음

#### TC-041: 스크롤 프레임 드랍
- **전제조건**: 채팅 메시지가 많음 (1000개 이상)
- **실행**:
  1. 채팅을 빠르게 스크롤
  2. 메뉴 열기/닫기
- **기대결과**: 
  - 스크롤이 60fps 유지
  - 프레임 드랍 없음
  - 메뉴 애니메이션이 부드러움

#### TC-042: 메모리 누수 체크
- **전제조건**: 채팅방 진입
- **실행**:
  1. 메뉴 열기/닫기 10회 반복
  2. 신고 모달 열기/닫기 10회 반복
  3. 차단 모달 열기/닫기 10회 반복
- **기대결과**: 
  - 메모리 사용량이 증가하지 않음
  - 이벤트 리스너가 정상적으로 cleanup됨
  - 콘솔에 경고 없음

### E2E 테스트 (Playwright / Cypress 권장)

#### E2E-001: 메뉴 열기/닫기
```javascript
// Playwright 예시
test('메뉴 열기/닫기', async ({ page }) => {
  await page.goto('/app/chat/chatId');
  await page.click('[aria-label="옵션 메뉴"]');
  await expect(page.locator('.menu-content')).toBeVisible();
  await page.click('[aria-label="옵션 메뉴"]');
  await expect(page.locator('.menu-content')).not.toBeVisible();
});
```

#### E2E-002: 신고 플로우
```javascript
test('신고 플로우', async ({ page }) => {
  await page.goto('/app/chat/chatId');
  await page.click('[aria-label="옵션 메뉴"]');
  await page.click('text=신고');
  await expect(page.locator('text=신고하시겠습니까?')).toBeVisible();
  await page.click('text=신고');
  await expect(page.locator('text=필요한 조치를 검토 중입니다.')).toBeVisible();
});
```

#### E2E-003: 차단 플로우
```javascript
test('차단 플로우', async ({ page }) => {

});
```

#### UNIT-002: 중복 요청 가드
```javascript
test('중복 신고 요청 차단', async () => {
  const { result } = renderHook(() => useReportUser());
  await act(async () => {
    await result.current.report();
  });
  expect(result.current.isReporting).toBe(true);
  
  // 두 번째 요청 시도
  await act(async () => {
    await result.current.report();
  });
  // 두 번째 요청은 무시되어야 함
  expect(mockApiCall).toHaveBeenCalledTimes(1);
});
```

```javascript
test('신고 API 에러 처리', async () => {

});
```
