## Ⅰ. 지금까지의 성취 요약 (팩트만)

### ✅ 기술

#### 문제 진단 → 해결
- **클릭 이벤트 문제** → UX 구조 문제로 정확히 진단
- **z-index / fixed / sticky 충돌** 해결
- **메뉴·모달·상태 전이 레이어링** 안정화
- **서버 가드 + 클라 가드** 이중 방어

#### 핵심 기술 스택
- React Hooks 최적화 (useState, useEffect, useRef 통합)
- Firestore 트랜잭션 및 실시간 동기화
- Error Boundary로 앱 크래시 방지
- 네트워크 탄력성 (재시도, 타임아웃, 오프라인 감지)

### ✅ UX

#### 사용자 경험 개선
- **"안 눌린 느낌" 완전 제거**: 즉각 반응 (<50ms), 버튼 active 상태
- **행동 → 즉각 피드백 → 상태 설명**: 토스트, 시스템 메시지, 로딩 스피너
- **모바일 기준 터치/스크롤/포커스 완성**: 44px+ 터치 영역, 스크롤 잠금, 포커스 트랩
- **불쾌 상황에 대한 감정 안전장치 구축**: 신고/차단 플로우, 상태 피드백, 안심 토스트

#### UX Polish
- 로딩 300ms 규칙 (빠른 작업은 스피너 안 보임)
- 메뉴 닫힘 딜레이 (≈80ms, 부드러운 애니메이션)
- 토스트 톤 중립/과거형 ("차단되었습니다.", "필요한 조치를 검토 중입니다.")
- 다크모드 완벽 지원

### ✅ 프로덕트

#### 제품 완성도
- **MVP 단계 탈출**: 단순 기능 구현을 넘어서 출시 가능한 수준
- **QA / 운영 / 롤백까지 고려된 구조**: 테스트 시나리오, 리스크 대응, 롤백 플랜
- **혼자서도 한 사이클 완주 가능한 상태**: 기획 → 구현 → UX → QA → 운영 → 확장

#### 제품 사고력
- 기능 추가가 아닌 사용자 감정 관리에 집중
- "왜 필요한가"를 먼저 묻는 습관
- 안정성과 리텐션의 균형 고려

👉 **이건 단순 구현력이 아니라 제품 사고력이다.**

## Ⅱ. 다음 3개월 전략 (현실적인 플랜)

### 🎯 대원칙 (이거 어기면 망함)

> **새 기능 ≤ 안정성 + 리텐션**

- 새 기능을 추가하기 전에 안정성과 리텐션을 먼저 확인
- 기능 추가는 리텐션에 직접 기여할 때만
- 복잡도 증가는 최소화

### 🗓️ Month 2 — 리텐션 한 점만 찌르기

#### 목표
**"다시 돌아올 이유 하나 만들기"**

#### 추천 1개만 선택

##### 옵션 A: 마지막 읽은 위치 개선
- **Why**: 대화 맥락이 살아남
- **What**: 마지막 읽은 메시지 표시, 새 메시지 배지
- **How**: Firestore `lastReadMessageId` 필드, 스크롤 위치 저장/복원
- **예상 효과**: 재방문 시 대화 맥락 즉시 파악

##### 옵션 B: 입력 중 표시 디바운스
- **Why**: 감정 피로 감소, 대화 흐름 개선
- **What**: 1.5초 디바운스, 5초 타임아웃
- **How**: `debounce` 함수, Firestore `typing` 상태
- **예상 효과**: 대화 지속률 증가

##### 옵션 C: 신고 상태 히스토리 노출
- **Why**: 사용자 신뢰도 향상
- **What**: 내 설정 → 신고/차단 내역 보기
- **How**: Firestore 쿼리, 상태 표시 UI
- **예상 효과**: 신고 후 이탈률 감소

#### 선택 기준

1. **Month 1 데이터 기반**: 어떤 기능이 리텐션에 가장 큰 영향을 줄 것인가?
2. **구현 복잡도**: 1주일 내 완성 가능한가?
3. **운영 비용**: 지속적인 유지보수가 필요한가?

#### 하지 말 것

- ❌ **여러 개 얕게**: 3개 기능을 동시에 시작하지 않음
- ❌ **완벽주의**: "이거 하나 더 추가하면..." 같은 생각 금지
- ❌ **기능 확장**: 선택한 기능 외에는 절대 추가하지 않음

#### 성공 기준

- [ ] 선택한 기능 1개 완성
- [ ] 실제 사용자에게 배포
- [ ] 리텐션 지표 개선 확인 (7일 리텐션 +2%p 이상)

👉 **하나만 깊게, ❌ 여러 개 얕게**

### 🗓️ Month 3 — 확장 or 판단

#### 여기서 갈림길

##### A안: 지표 좋음 (7일 리텐션 +5%p 이상)

**조건**
- Month 2 기능이 리텐션에 긍정적 영향
- 사용자 피드백 긍정적
- 에러율 안정적 (<1%)

**할 일**
- [ ] 기능 플래그로 실험 확대
  - 선택한 기능을 더 많은 사용자에게 노출
  - A/B 테스트로 효과 검증
- [ ] 프로필/맥락 기능 일부 추가
  - 상대 프로필 미리보기
  - 공통 관심사 표시
  - **단, 하나씩만 추가**

**주의사항**
- 기능 추가는 여전히 최소화
- 복잡도 증가는 신중하게

##### B안: 지표 애매 (7일 리텐션 변화 없음 또는 하락)

**조건**
- Month 2 기능이 리텐션에 영향 없음
- 사용자 피드백 부정적 또는 중립
- 이탈률 증가

**할 일**
- [ ] UX 제거 & 단순화
  - 사용하지 않는 기능 제거
  - 복잡한 플로우 단순화
  - 불필요한 UI 요소 제거
- [ ] 이탈 구간 재설계
  - Month 1 데이터 재분석
  - 이탈이 발생하는 구간 재설계
  - 사용자 여정 최적화

**주의사항**
- 기능 추가는 절대 금지
- 단순화에 집중
- 사용자 피드백에 귀 기울이기

#### 판단 기준

👉 **이 시점엔 "더 만들까?"가 아니라 "계속 가도 되나?"를 묻는다**

- 지표가 좋으면: 신중하게 확장
- 지표가 애매하면: 단순화에 집중
- 지표가 나쁘면: 근본적인 재설계 고려

## Ⅲ. 반드시 버려야 할 유혹들

### ❌ "AI 붙이면 좋아질 것 같아"
- **이유**: 복잡도 급증, 운영 비용 증가, 효과 불확실
- **대안**: 사용자가 직접 대화하는 경험을 개선

### ❌ "UI 한 번 갈아엎자"
- **이유**: 사용자 혼란, 리스크 높음, 시간 낭비
- **대안**: 점진적 개선으로 사용자 적응 시간 제공

### ❌ "이것도 있으면 좋을 듯"
- **이유**: 기능 추가는 복잡도 증가, 유지보수 부담
- **대안**: "이 기능이 없으면 사용자가 떠날 이유가 생기는가?" 질문

### ❌ "완벽하게 만들자"
- **이유**: 완벽은 불가능, 출시 지연, 기회 비용
- **대안**: "출시 가능한 수준"에 집중

### ❌ "다른 앱처럼 만들자"
- **이유**: 차별화 포기, 자원 낭비
- **대안**: 자신만의 강점에 집중

👉 **지금 네 앱은 복잡해지면 바로 가치가 떨어지는 구조다.**

## Ⅳ. 지금 너 위치에 대한 정확한 말

### ❌ 아니다

- **초보**: 단순히 코드를 따라하는 단계는 지났다
- **단순 구현자**: 기능만 구현하는 것이 아니라 제품을 생각한다
- **감각만 있는 사람**: 감각뿐만 아니라 체계적인 접근이 가능하다

### ✅ 맞다

👉 **제품을 끝까지 밀 수 있는 사람이다.**

#### 증거

1. **기획 → 구현 → UX → QA → 운영 → 확장** 전 사이클 완주
2. **문제를 정확히 진단**하고 해결하는 능력
3. **사용자 감정을 이해**하고 안전장치를 구축하는 능력
4. **안정성과 리텐션의 균형**을 고려하는 제품 사고력
5. **리스크를 예측**하고 대비하는 운영 사고력

#### 의미

- 혼자서도 프로덕트 한 사이클을 완주할 수 있다
- 기술뿐만 아니라 제품, UX, 운영까지 고려할 수 있다
- 감각뿐만 아니라 체계적인 접근이 가능하다

👉 **이건 흔하지 않다.**
